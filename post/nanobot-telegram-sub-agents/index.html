nanobot telegram sub-agents

在 nanobot 中为 Telegram 聊天/群组/频道指定专属 Agent 步骤指南

目标

让 nanobot 在指定的 Telegram 聊天（私聊、群组或频道）中只运行一个高度专注的 Agent，专门处理特定事务（例如 RSS 摘要），不闲聊、不越界。

步骤概览

  1. 创建专属工作目录

  2. 编写行为定义文件（类似 SOUL.md）

  3. 链接所需技能

  4. 创建 Agent 并设置身份

  5. 绑定 Telegram 聊天/群组/频道

  6. 重启服务并验证

1. 创建专属工作目录

为专用 Agent 建立独立的工作空间，存放其人格设定和可用技能。

bash

bash
mkdir -p ~/.nanobot/workspace-rss/skills

将 workspace-rss 替换为实际用途的名称，例如 workspace-weather。

2. 定义行为边界（核心）

在 workspace-rss 目录下创建 SOUL.md（或其他 nanobot 规定的行为配置文件），明确 Agent 的职责和限制。
示例 SOUL.md：

markdown

markdown
# SOUL.md - RSS 摘要专用 Agent

## 核心原则
你是一个 RSS 内容摘要助手。唯一职责是按照用户要求，使用 `miniflux-reader` 技能获取和总结 Miniflux 中的订阅内容。

## 行为规则
- **严格执行**：用户要求总结几条就几条，指定哪个 feed/category 就总结哪个。
- **不扩展**：不主动添加用户未要求的内容。
- **不闲聊**：不寒暄、不发表与 RSS 无关的评论。
- **标记已读**：仅在用户明确要求“总结新的”时，才将上一轮总结过的条目标记为已读；仅查询时不标记。

## 边界
- 拒绝处理与 RSS/Feed 无关的请求，礼貌说明只负责 RSS 内容摘要。

通过明确“不做什么”让 Agent 保持专注。

3. 链接所需技能

将需要用到的技能软链接到工作区的 skills 目录。假设技能仓库在 ~/.nanobot/skills/。

bash

bash
ln -s ~/.nanobot/skills/miniflux-reader ~/.nanobot/workspace-rss/skills/miniflux-reader

  • 软链接便于技能更新时自动同步。

  • 如需多个技能，依次链接。

4. 创建 Agent 并设置身份

使用 nanobot 命令行工具创建 Agent，指定工作区和模型。

bash

sql
nanobot agents add rss \
  --workspace ~/.nanobot/workspace-rss \
  --model your-preferred-model  # 例如 openai/gpt-4, claude-3 等

然后设置身份（名称和表情符号）：

bash

arduino
nanobot agents set-identity --agent rss --name "RSS 助手" --emoji "📰"

  • 身份信息会显示在回复中，方便区分是哪个 Agent 在响应。

5. 绑定 Telegram 聊天/群组/频道

编辑 nanobot 的主配置文件（通常为 ~/.nanobot/nanobot.json），在 bindings 数组中添加一条路由规则，将特定 Telegram 实体的消息指向刚创建的 Agent。

获取 Telegram 聊天/群组/频道 ID：

  • 向 @getidsbot 发送任意消息，它会返回当前聊天的 ID 等信息。

  • 或者在群组/频道中，将机器人添加为管理员，然后通过机器人 API 获取更新中的 chat.id。

示例配置片段（假设 nanobot 使用 "channel": "telegram" 标识平台）：

json

json
{
  "bindings": [
    // 其他绑定规则...
    {
      "agentId": "rss",
      "match": {
        "channel": "telegram",
        "peer": {
          "kind": "chat",        // 可选值：chat（私聊）、group、channel
          "id": 123456789        // 替换为你的聊天/群组/频道 ID（整数）
        }
      }
    }
  ]
}

  • 私聊："kind": "chat"

  • 群组："kind": "group"

  • 频道："kind": "channel"

如果 nanobot 支持按整个群组/频道路由而不区分具体 ID，可能还有 groupId、channelId 字段，请根据实际文档调整。

路由匹配维度参考（根据 nanobot 实际支持情况调整）：

字段用途peer.kind + id匹配特定私聊、群组或频道chatId匹配所有来自该 ID 的聊天（简写）accountId匹配特定 Telegram 账号（多账号场景）

保存文件后，重启 nanobot 的网关服务以使配置生效：

bash

auto

6. 验证

在绑定的 Telegram 聊天中发送一条简单消息，例如“你是谁”。

  • 如果回复中包含你设置的身份（名称或表情符号），说明路由成功。

  • 也可以尝试发一条 RSS 摘要指令，观察是否只调用 miniflux-reader 技能且不闲聊。

注意事项

  • 备份配置：修改 nanobot.json 前建议备份原文件。

  • 模型选择：专用 Agent 可使用较小或成本较低的模型，因为任务单一。

  • 技能路径：根据你的技能安装位置调整软链接命令。

  • 多 Agent 共存：可以创建多个专用 Agent，通过不同的路由规则分配到不同的 Telegram 聊天。

通过以上步骤，你就能在 nanobot 中为 Telegram 实现专属 Agent，让每个聊天拥有一个高度专注的助手。

为 Discord 频道指定专属 agent 实现专注高效处理事务

OpenClaw 默认只有一个主 agent，所有频道的消息都由它处理。但有些场景需要一个更专注的 agent：只做特定的事，不闲聊，不擅自扩展。

OpenClaw 的 agent 系统支持创建独立的 agent，每个 agent 有自己的工作目录、人格设定和 skill 配置，然后通过路由绑定把特定频道的消息分发给对应的 agent。举个例子，在 Discord 上建一个 RSS 频道，专门用来总结 Miniflux 里的订阅内容。下面记录一下这个配置过程。

创建专用工作目录

每个 agent 需要一个独立的工作目录，里面放它的人格设定和可用的 skill。

auto
bash
mkdir -p ~/.openclaw/workspace-rss/skills

SOUL.md：定义行为边界

SOUL.md 是 agent 的核心，决定了它"是谁"、“能做什么”、“不能做什么”。专用 agent 的关键是把边界画清楚：

auto
markdown
# SOUL.md - RSS 总结专用 Agent

## 核心原则

你是一个 RSS 内容总结专用助手。你的唯一职责是按照用户的要求，
使用 miniflux-reader skill 获取和总结 Miniflux 中的订阅内容。

## 行为规则

**严格执行，不多不少：**
- 用户让总结几条就总结几条，让总结哪个 feed/category 就总结哪个
- 不主动扩展、不添加用户没要求的内容
- 不闲聊、不寒暄、不发表与 RSS 总结无关的评论

**标记已读规则：**
- 只有在用户明确要求总结新的分类/订阅源时，才把上一轮总结过的条目标记为已读
- 用户只是查询信息、没有要求总结新内容时，不执行标记

## 边界

- 不处理与 RSS/Feed 无关的请求
- 遇到无关请求时，礼貌地说明自己只负责 RSS 内容总结

和主 agent 的 SOUL.md 对比，专用 agent 多了很多"不"——不闲聊、不扩展、不越界。这是故意的。通用 agent 需要灵活，专用 agent 需要克制。

链接 Skill

把需要的 skill 软链接到工作目录的 skills 目录：

auto
bash
ln -s ~/.claude/skills/miniflux-reader ~/.openclaw/workspace-rss/skills/miniflux-reader

这样 agent 就能使用 miniflux-reader skill 来获取和处理 Miniflux 的未读条目。用软链接而不是复制，好处是 skill 更新时不用同步。

其他文件

IDENTITY.md 定义名称和 emoji，用来区分是哪个 agent 在回复：

auto
markdown
- **Name:** RSS 助手
- **Emoji:** 📰

USER.md 从主工作目录复制过来就行，同一个用户。

创建 Agent

用 openclaw agents add 创建 agent：

auto
bash
openclaw agents add rss \
  --workspace ~/.openclaw/workspace-rss \
  --model bailian/glm-5 \

--model 参数改成你自己要用的模型。

设置 identity：

auto
bash
openclaw agents set-identity --agent rss --name "RSS 助手" --emoji "📰"

绑定 Discord 频道

这一步是把 Discord 频道的消息路由到专用 agent。需要直接编辑 ~/.openclaw/openclaw.json，在现有的 bindings 数组中追加一条绑定规则（不要覆盖整个文件）：

auto
json
{
  "agentId": "rss",
  "match": {
    "channel": "discord",
    "peer": {
      "kind": "channel",
      "id": "<你的频道ID>"
    }
  }
}

频道 ID 的获取方式：在 Discord 中开启开发者模式（设置 → 高级 → 开发者模式），然后右键频道名 → 复制频道 ID。

改完后重启 gateway：

auto
bash
openclaw gateway restart

用 openclaw agents bindings 验证，应该能看到：

ini
- rss <- discord peer=channel:<你的频道ID>

OpenClaw 的路由匹配支持几种维度：

字段用途peer.kind + peer.id匹配特定频道、群组或 DMguildId匹配整个 Discord 服务器accountId匹配特定账号（多账号场景）roles按 Discord 角色路由

按频道路由用 peer，按服务器路由用 guildId，按用户路由用 accountId。

验证

最直观的方式是在 Discord 频道里问"你是谁"。如果回答的是专用 agent 的名字而不是主 agent 的名字，说明路由生效了。
