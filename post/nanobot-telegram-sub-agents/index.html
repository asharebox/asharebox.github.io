nanobot telegram sub-agents

在 nanobot 中为 Telegram 聊天/群组/频道指定专属 Agent 步骤指南

目标

让 nanobot 在指定的 Telegram 聊天（私聊、群组或频道）中只运行一个高度专注的 Agent，专门处理特定事务（例如 RSS 摘要），不闲聊、不越界。

步骤概览

  1. 创建专属工作目录

  2. 编写行为定义文件（类似 SOUL.md）

  3. 链接所需技能

  4. 创建 Agent 并设置身份

  5. 绑定 Telegram 聊天/群组/频道

  6. 重启服务并验证

1. 创建专属工作目录

为专用 Agent 建立独立的工作空间，存放其人格设定和可用技能。

bash

bash
mkdir -p ~/.nanobot/workspace-rss/skills

将 workspace-rss 替换为实际用途的名称，例如 workspace-weather。

2. 定义行为边界（核心）

在 workspace-rss 目录下创建 SOUL.md（或其他 nanobot 规定的行为配置文件），明确 Agent 的职责和限制。
示例 SOUL.md：

markdown

markdown
# SOUL.md - RSS 摘要专用 Agent

## 核心原则
你是一个 RSS 内容摘要助手。唯一职责是按照用户要求，使用 `miniflux-reader` 技能获取和总结 Miniflux 中的订阅内容。

## 行为规则
- **严格执行**：用户要求总结几条就几条，指定哪个 feed/category 就总结哪个。
- **不扩展**：不主动添加用户未要求的内容。
- **不闲聊**：不寒暄、不发表与 RSS 无关的评论。
- **标记已读**：仅在用户明确要求“总结新的”时，才将上一轮总结过的条目标记为已读；仅查询时不标记。

## 边界
- 拒绝处理与 RSS/Feed 无关的请求，礼貌说明只负责 RSS 内容摘要。

通过明确“不做什么”让 Agent 保持专注。

3. 链接所需技能

将需要用到的技能软链接到工作区的 skills 目录。假设技能仓库在 ~/.nanobot/skills/。

bash

bash
ln -s ~/.nanobot/skills/miniflux-reader ~/.nanobot/workspace-rss/skills/miniflux-reader

  • 软链接便于技能更新时自动同步。

  • 如需多个技能，依次链接。

4. 创建 Agent 并设置身份

使用 nanobot 命令行工具创建 Agent，指定工作区和模型。

bash

sql
nanobot agents add rss \
  --workspace ~/.nanobot/workspace-rss \
  --model your-preferred-model  # 例如 openai/gpt-4, claude-3 等

然后设置身份（名称和表情符号）：

bash

arduino
nanobot agents set-identity --agent rss --name "RSS 助手" --emoji "📰"

  • 身份信息会显示在回复中，方便区分是哪个 Agent 在响应。

5. 绑定 Telegram 聊天/群组/频道

编辑 nanobot 的主配置文件（通常为 ~/.nanobot/nanobot.json），在 bindings 数组中添加一条路由规则，将特定 Telegram 实体的消息指向刚创建的 Agent。

获取 Telegram 聊天/群组/频道 ID：

  • 向 @getidsbot 发送任意消息，它会返回当前聊天的 ID 等信息。

  • 或者在群组/频道中，将机器人添加为管理员，然后通过机器人 API 获取更新中的 chat.id。

示例配置片段（假设 nanobot 使用 "channel": "telegram" 标识平台）：

json

json
{
  "bindings": [
    // 其他绑定规则...
    {
      "agentId": "rss",
      "match": {
        "channel": "telegram",
        "peer": {
          "kind": "chat",        // 可选值：chat（私聊）、group、channel
          "id": 123456789        // 替换为你的聊天/群组/频道 ID（整数）
        }
      }
    }
  ]
}

  • 私聊："kind": "chat"

  • 群组："kind": "group"

  • 频道："kind": "channel"

如果 nanobot 支持按整个群组/频道路由而不区分具体 ID，可能还有 groupId、channelId 字段，请根据实际文档调整。

路由匹配维度参考（根据 nanobot 实际支持情况调整）：

字段用途peer.kind + id匹配特定私聊、群组或频道chatId匹配所有来自该 ID 的聊天（简写）accountId匹配特定 Telegram 账号（多账号场景）

保存文件后，重启 nanobot 的网关服务以使配置生效：

bash

auto

6. 验证

在绑定的 Telegram 聊天中发送一条简单消息，例如“你是谁”。

  • 如果回复中包含你设置的身份（名称或表情符号），说明路由成功。

  • 也可以尝试发一条 RSS 摘要指令，观察是否只调用 miniflux-reader 技能且不闲聊。

注意事项

  • 备份配置：修改 nanobot.json 前建议备份原文件。

  • 模型选择：专用 Agent 可使用较小或成本较低的模型，因为任务单一。

  • 技能路径：根据你的技能安装位置调整软链接命令。

  • 多 Agent 共存：可以创建多个专用 Agent，通过不同的路由规则分配到不同的 Telegram 聊天。

通过以上步骤，你就能在 nanobot 中为 Telegram 实现专属 Agent，让每个聊天拥有一个高度专注的助手。
